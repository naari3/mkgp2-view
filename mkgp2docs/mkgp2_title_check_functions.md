# MKGP2 称号チェック関数テーブル解析

注意！！！内容が大きく違うことが分かっており、[v2版](./mkgp2_title_check_functions_v2.md)を新たに作成しました。そちらを参照してください。

## 概要

`determine_title` (0x800a70fc) がモードに応じたチェック関数テーブルを順にイテレーションし、
最初にマッチした関数が `DAT_806d13d4` (staging TitleID) と `DAT_806d13d8` (param) を書き込む。
テーブルは先頭(index 0)が最優先、末尾が最低優先。

**重要: Fixed テーブルの TitleID オフセット (+0x2D)**

determine_title の Fixed テーブルループ内で、check 関数が返す raw TitleID に **+0x2D (45)** が加算される
(PPC: `addi r5, r3, 0x2d` at 0x800a7394)。Extra 関数の TitleID にはこのオフセットは適用されない。

- **Extra 関数**: 直接 TitleID = raw 値 (TitleID 1-45: GP制覇系)
- **Fixed テーブル**: 最終 TitleID = raw 値 + 45

以下の各関数説明の TitleID は **最終値 (raw + 45 適用済み)** で記載している。

**重要: 同一称号の重複回避メカニズム**

Fixed テーブルループで check 関数がマッチしても、算出された TitleID が `g_currentTitleId` と
同じ場合はスキップされ、ループが次のエントリに進む（例外: TitleID 0x42=66 のみ重複許可）。
これにより、プレイのたびに異なる称号が付与される。

```c
// DetermineTitle 内の Fixed テーブルループ条件
if (fixedCheckPassed && g_stagingTitleId > 0) {
    g_stagingTitleId += 0x2d;
    if (g_stagingTitleId != g_currentTitleId || g_stagingTitleId == 0x42) {
        break;  // 採用
    }
    // else: 同じ称号 → スキップして次へ
}
```

**実動作の観察結果**:
- カード作成直後: TitleID なし (ただの「ルーキー」)。g_playCount は DetermineTitle 呼び出し前に
  インクリメント済みのため、Race[0] (初回プレイチェック: g_playCount==0) は通常マッチしない。
- 初レース後 (150cc): Race[22] FUN_800a7e38 がヒット → 「超一流のルーキー」(TitleID 136)
- 初レース後 (100cc): Race[23] FUN_800a7e0c がヒット → 「パーフェクトルーキー」(TitleID 137)
- g_currentTitleId=114 (ゴールデン) の場合: Race[21] FUN_800a7e64 がヒット → 「通りすがりのルーキー」(TitleID 135)

## 優先度の仕組み

テーブルは **index 0 が最優先** で順に走査される。最初にマッチした関数のTitleIDが採用される。

例 (Race):
1. まず「初回プレイか？」(index 0) → YES なら "さまよえる" (153) で確定
2. NO なら「走行距離が特定範囲か？」(index 1) → YES なら距離系称号 (105-108)
3. NO なら「バトル勝利数がテーブル値か？」(index 2) → ...
4. ...
5. 全部NOなら番兵 (index 31) が return 0 → 称号変更なし

## 備考

- グローバル変数名は全て推定。実機テストで確認が必要
- FUN_800a7600 (Race Extra) と FUN_800a7548 (Battle Extra) は Fixed テーブル走査前に呼ばれる別枠
  - Extra 関数は +0x2D オフセットなしで TitleID 1-45 (GP制覇/カップクリア系) を直接設定
- FUN_800a8a28 (NameID決定) は別の解析が必要
- +0x2D オフセット適用により、チェック条件とタイトル名の対応は概ね直感的に理解できる
  - 例: 3勝0敗 → 無敗, 敗北復帰 → 元無敗, 連勝 → 連勝バトル
- 称号 ID 91-93 (136-138) については `g_technicalTitleGrade` の変遷が鍵となる。 [詳細調査結果](./mkgp2_analysis_technical_flag.md)

## NameID 決定ロジック (determine_name_id / FUN_800a8a28)

### 概要

DetermineTitle から TitleID 決定**前**に呼ばれ、プレイヤーの進行度スコアに基づいて
NameID (称号の名詞部分: ルーキー、レーサー、キング等) を決定する。
TitleID とは独立した計算で、TitleID を入力としない。

### 呼び出しコンテキスト (DetermineTitle 内)

```c
// ゴーストクリア数カウント (表/裏 × 8コース × 8ゴースト = 最大128)
clearedCount = 0;
for (i = 0; i < 2; i++)           // 表/裏
  for (course = 1; course < 9; course++)  // 8コース
    for (ghost = 0; ghost < 8; ghost++)   // 8ゴースト
      if (FUN_801d5b8c(player, i, course, ghost)) clearedCount++;

// コースクリア数カウント (8コース × 表/裏 × 4点 = 最大64)
for (course = 1; course < 9; course++)
  for (mode = 0; mode < 2; mode++)
    if (FUN_801d59f8(player, course, mode)) clearedCount += 4;

g_nameId = determine_name_id(clearedCount + player_wins, character_id);
```

### パラメータ

| パラメータ | 意味 | 計算 |
|-----------|------|------|
| param_1 (score) | 進行度スコア | ゴーストクリア数 + コースクリア数×4 + 勝利数 |
| param_2 (character_id) | キャラクターID (DAT_806d125c) | 値3,9 = 女性キャラ (ヒロイン/プリンセス/クイーン分岐) |

### 内部処理

関数内部でも FUN_801d59f8 を呼んで**表/裏の全コースクリア状態**を個別カウント:
- r27 = 表コースクリア数 (0-8)
- r26 = 裏コースクリア数 (0-8)

### 判定優先順位

#### 1. 裏全クリア + 高スコア → 最強伝説

- **条件**: 裏クリア数 == 8 AND score >= 150
- **NameID**: 37 (最強伝説 / Legendary)

#### 2. 表全クリア → マリカ鬼神 [バグあり]

- **条件**: 表クリア数 == 8
- **NameID**: score ≤ 128 → 36 (マリカ鬼神 / MK Lord)
- **NameID**: score > 128 → メイン本体にフォールスルー

**バグ**: PPC アセンブリで閾値が**降順** (128, 124, 120, ..., 100) にチェックされるため、
最初の比較 (score ≤ 128) が該当する全スコアをキャッチし、**NameID 29-35 は到達不可能**。
本来は昇順 (100, 104, ..., 128) で NameID 29→30→...→36 と段階的に上がるはずだった。
